1. 评论模块 (Comment) - 【重要不一致】
- 接口文档 (7.4 发表评论 / 留言): 请求体中定义了 `website` (个人网站地址) 字段，允许用户在评论时填写个人网站。
- 建表语句 (`comment` 表): 结构中完全缺失了 `website` 字段。这将导致前端传过来的 website 数据无法持久化保存到数据库中。

2. 文章模块 (Article) - 【潜在的入参遗漏】
- 建表语句 (`article` 表): 表结构中设计了用于 SEO 的 `slug` (URL别名) 和 `keywords` (SEO关键词) 字段。
- 接口文档 (5.1 创建文章 & 5.3 更新文章): 请求体 (JSON) 参数中没有包含 `slug` 和 `keywords`。
  * 建议：如果这两个字段需要由作者手动设置，请在接口文档的入参中补充；如果是后端自动生成（例如根据标题生成拼音slug，提取内容生成keywords），则可以在文档中备注说明。

3. 操作日志模块 (Operation Log) - 【字段命名映射差异】
接口文档 (14.2 查看操作日志) 约定的返回字段与建表语句 (`sys_oper_log`) 的字段名存在较大差异。虽然在代码层可以通过实体类 (DTO) 进行别名映射，但为了规范和可读性，建议统一：
- 接口返回值 `module`   <--> 数据表字段 `title`
- 接口返回值 `operation`<--> 数据表字段无直接对应 (表中有 `business_type` 和 `method`，需明确映射关系)
- 接口返回值 `operator` <--> 数据表字段 `oper_name`
- 接口返回值 `ip`       <--> 数据表字段 `oper_ip`
- 接口返回值 `createTime`<--> 数据表字段 `oper_time` (注意表中并无 `create_time` 字段)

4. 角色权限模块 (System Role / Permission) - 【规范建议】
- 接口文档 (9.2 创建权限): 指出 `status` (状态：1-启用，0-禁用) 默认值为 1。
- 建表语句 (`sys_permission` 表): 确实有 `status tinyint not null default 1`，一致。但角色表 (`sys_role`) 的 `status` 字段缺省了 default 1 （SQL中为 `status tinyint not null default 1`），文档与SQL行为一致，此处设计合理，无需大改，仅作为核对确认。

5. 评论模块 - 缺失 user_agent 字段支持
问题描述：
建表语句：comment 表中设计了 user_agent varchar(500) 字段，用于记录用户评论时的设备信息（浏览器/手机型号）。
接口文档：在 7.4 发表评论 的请求体中未定义该字段，且未说明后端会自动从 HTTP Header (User-Agent) 中提取。
风险：如果后端代码依赖手动传参而非自动提取，该字段将永远为空；如果前端试图传参但文档没写，会造成联调困惑。
建议：在文档 7.4 的说明中补充：“user_agent 字段无需前端传递，后端将自动从请求头中提取并保存”。
6. 文章点赞模块 - 潜在的并发冲突
问题描述：
建表语句：article_like 表建立了联合唯一索引 uk_article_ip (article_id, ip_address)，防止同一 IP 重复点赞。
接口文档：5.12 文章点赞 的成功响应直接返回了新的点赞数。
缺失的错误码：文档的“错误码一览表”和 5.12 接口的失败响应中，缺少针对“重复点赞”的特定错误码。
目前只有通用的 1006 DATA_ALREADY_EXISTS，但业务上“重复点赞”通常视为成功操作（即忽略本次请求，直接返回当前最新数值），而不是报错。
建议：
明确 5.12 的逻辑：如果是重复点赞，是返回 409 Conflict 还是返回 200 OK (带当前计数)？
推荐做法：后端捕获唯一索引冲突，视为“已点赞”，直接返回 200 OK 和当前的 likeCount，前端无感知。需在文档中明确此行为，避免前端误判为失败。

7. 操作日志模块 (Operation Log) - 【查询参数与表结构不匹配】
问题描述：
接口文档 (14.2 查看操作日志)：查询参数中定义了 module (模块名称) 用于筛选。
建表语句 (sys_oper_log 表)：表中对应的字段是 title (模块标题)，没有 module 字段。
差异：虽然你在第 3 点提到了返回值的映射差异，但这里不仅是返回值，查询入参也存在不匹配。前端传 module=文章管理，后端 SQL 若直接拼接 WHERE module = ? 会报错；若映射为 WHERE title = ?，则文档需明确说明这种映射关系，否则前端开发者会困惑为何按“模块”搜不到数据。
建议：统一术语。要么将数据库字段 title 改为 module，要么在接口文档的查询参数说明中备注：“参数 module 对应数据库字段 title”。
8. 访问统计模块 (Visit Log) - 【文档缺失】
问题描述：
建表语句：设计了 visit_log 表，包含 ip_address, user_agent, page_url, referer 等详细字段，显然用于记录用户访问轨迹。
接口文档：完全缺失 关于 visit_log 表的任何接口定义（如：如何写入日志、后台如何查看访问明细列表）。
现状：文档第 11 节“站点统计”只提供了聚合数据接口（如趋势图、热门页面），但没有提供基于 visit_log 表的原始日志查询接口。
风险：visit_log 表的数据可能无法被有效利用或查看，沦为“死数据”；或者后端有隐式接口但未文档化，导致前后端联调困难。
建议：补充 GET /api/admin/visit-log/page 接口文档，用于在后台查看详细的访问流水记录。
